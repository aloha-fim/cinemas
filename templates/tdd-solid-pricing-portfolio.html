S

Single Responsibility

"A class should have only ONE reason to change"

The Problem

When a class does multiple jobs, any change to one job risks breaking the others. A "God Class" that handles pricing, discounts, fees, and tax is a nightmare to maintain.

The Solution

Split responsibilities into separate classes. BasePriceCalculator only handles base prices. TaxCalculator only handles tax. If tax logic changes, only one class changes.

Signs of Violation
  • Class has multiple unrelated methods
  • Changes to one feature break another
  • Class name includes "Manager" or "Handler"

O

Open/Closed

"Open for extension, closed for modification"

The Problem

Every time you need a new discount type, you modify the same switch/if-else block. This risks breaking existing discounts and requires re-testing everything.

The Solution

Define a Discount interface. New discounts are new classes that implement this interface. The pricing engine works with ANY discount without modification.

Signs of Violation
  • if/elif chains that grow with each new type
  • Switch statements on type codes
  • Adding features requires changing tested code

# Define interface once class Discount(ABC): def apply(self, price): ... # Add new discounts as NEW classes class StudentDiscount(Discount): ... class MilitaryDiscount(Discount): ... # Engine never changes! class PricingEngine: def add_discount(self, d: Discount): self.discounts.append(d)
L

Liskov Substitution

"Subtypes must be substitutable for base types"

The Problem

If a subclass behaves differently than expected (throws exceptions, returns wrong types), code that works with the base class will break when given the subclass.

The Solution

  • Subclass throws exception for inherited method

  • Subclass returns unexpected types

I

Interface Segregation

"Clients shouldn't depend on methods they don't use"

The Problem

A "fat" interface forces classes to implement methods they don't need. If Discount requires is_expired(), permanent discounts have to implement a meaningless method.

The Solution

Create focused interfaces: Applicable (all discounts), Stackable (some discounts), Expirable (promo codes only). Each class implements only what it needs.

Signs of Violation
  • Methods that throw "NotImplemented"
  • Empty method implementations
  • Classes with methods they don't actually use

D

Dependency Inversion

"Depend on abstractions, not concretions"

The Problem

When a class creates its own dependencies internally, you can't test it in isolation or swap implementations. Testing requires the real database, real API, etc.

The Solution

Inject dependencies via constructor. TicketPricingService accepts PriceProvider, TaxProvider, etc. In tests, inject mocks. In production, inject real implementations.

Signs of Violation
  • new keyword inside methods
  • Can't test without real database/API
  • Hard to swap implementations
